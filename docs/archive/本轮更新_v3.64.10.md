# 更新说明 v3.64.10 — 真正修好"放大字号后整页空白"

v3.64.9 把"放大后没字"往"重排 + 重建翻页"的方向修了，方向对，但只对了一半——你换包后实测：前后柔软翻页 OK 了，一放大还是整页空白（页码却仍显示 3/243）。这版找到并修掉真正的卡点。

## 根因（在 page-flip 库的初始化时序）

读了库源码确认：`loadFromHTML()` 内部是这个顺序——

1. 同步 `pages.show(startPage)`：先把"起始页"显示出来（不指定就是第 0 页）。
2. **`setTimeout(0)` 里才跑 `ui.update()`**：定稿真正的页面布局 / 页位 / 画布尺寸——这是晚一拍的。

库给每个页元素的默认样式是 `display:none`，靠 `pages.show()` 配合 `ui.update()` 定稿的位置把当前页显示出来。

重建翻页时（改字号、或窗口缩放触发），我们紧接着 `loadFromHTML` 调用的 `turnToPage(当前页)` **赶在那个 `setTimeout(0)` 的 `ui.update()` 之前**执行了。结果当前页在布局还没定稿时被摆错位 / 仍是 `display:none`，整页就空了——而我们自己维护的页码变量没变，所以底部仍显示"3 / 243 · 1%"。

首次建翻页（`loadBook` 里）之所以没事，是因为它用了双 `requestAnimationFrame` 等布局完全稳定后才建；重建路径（resize 和上一版新增的字号重排）却是**同步**建，正好踩中这个时序坑。

## 改法（三处）

1. **建实例时 `startPage = 当前页`**：让库自身那次同步的 `pages.show(startPage)` 一开始就摆当前页，而不是先摆第 0 页再被我们纠正。
2. **`loadFromHTML` 后再补定位**：在库的 `setTimeout(0)`（`ui.update` 定稿）之后、以及再下一帧，各补一次 `turnToPage(当前页)`，确保布局定稿后当前页确实显示出来。
3. **重建统一改为延迟 + 核验**：字号 / 字体 / 行距改动、以及窗口 resize，触发的重建都走同一个 `rebuildFlipDeferred()`——双 `requestAnimationFrame` 等布局稳定再 `initFlip()`，建完再核验一眼当前页是不是真渲染出了文字，万一还空就再补建一次。与首次建翻页同款时序，不再同步重建。

## 改动文件

- `frontend/reader/reader.js`：
  - `tryInitStPageFlip`：新增 `startPage`；`loadFromHTML` 后用 `setTimeout(0)` + `requestAnimationFrame` 各补一次 `turnToPage`。
  - 新增 `rebuildFlipDeferred()`（双 rAF + 空白核验补建）；`repaginate()` 与 resize 处理改为调用它，不再同步 `initFlip()`。

（向前 / 向后柔软翻页沿用 v3.64.9 的修复——关掉库的"书角判断"，已确认两向正常。收藏翻阅的柔和翻页也保持 v3.64.9。）
