# 更新说明 v3.64.11 — 真正修好"小窗最大化后整页空白"

你那两张截图把问题说死了：小窗里有字（截图一），一点最大化整页变空、页码还冻在 1/843 不变（截图二）。前几版我一直以为是"重建时序"的问题，方向偏了。这版读了 page-flip 库的源码，找到真正的机制并修掉。

## 真正的根因（在 page-flip 库里）

读库源码确认两点：

- `setHandlers()` 里**无条件**执行 `window.addEventListener("resize", this.onResize)`——每建一次翻页实例就给 window 挂一个 resize 监听。
- 但 `destroy()` 只有在 `useMouseEvents` 为真时才调用 `removeHandlers()`（移除监听的那段就在 removeHandlers 里）：

  ```
  destroy(){ this.app.getSettings().useMouseEvents && this.removeHandlers(), ... }
  ```

我们为了"点书页不乱翻、能选词划线"，把 `useMouseEvents` 设成了 `false`。结果：**那个 resize 监听永远不会被移除**，一路累积；而且当前这个活实例的 `onResize` 会在窗口尺寸变化时触发。

它的 `onResize → render.update → calculateBoundsRect` 用的是"**当前（放大后）的容器尺寸**"和"**建实例时定下的固定页尺寸**（`size:'fixed'`）"去重算页面位置。最大化后两者对不上，页面被算到可视区域之外 → 整页空白；而页码是我们自己维护的变量，没参与这次错误重排，所以仍显示 1/843，看起来像"冻住"。

这也解释了为什么前几版没修好：我改的是"我们自己重建翻页"的时序，但真正捣乱的是**库自带、且不会被清理的那个 resize 监听**，它在我们重建之前就把页面弄空了。

## 修法（两处，互相配合）

1. **`destroyFlip()` 在 `destroy()` 前临时把 `useMouseEvents` 置真**：逼库走 `removeHandlers()` 把它注册的 resize 监听摘掉（顺带要移除的鼠标/触摸监听本来就没注册，`removeEventListener` 对不存在的监听无害）。这样每次销毁都不会残留监听，不再累积。

2. **我们自己的 `resize` 监听里先 `destroyFlip()`**：我们这个监听是在脚本初始化时注册的，**早于**库在 `loadFromHTML` 里注册的 `onResize`。所以同一次 resize 事件里我们先执行；`destroyFlip` 把库的 resize 监听摘掉——按 DOM 规范，在事件派发过程中移除一个尚未被调用的监听，它本次就不会再被调用。于是库的 `onResize` 这次根本不会执行，最大化时不再"挪位 → 空白"。随后照旧防抖按新尺寸重建一个新实例（仍带 `startPage`=当前页、双 rAF 等布局稳、建完核验没字就补建的那套保险）。

## 改动文件

- `frontend/reader/reader.js`：
  - `destroyFlip()`：`destroy()` 前 `pageFlip.getSettings().useMouseEvents = true`，强制库清掉自己的 resize 监听。
  - `window 'resize'` 处理：进来先 `destroyFlip()`（趁库的 `onResize` 还没轮到就把它摘掉），再防抖重建。
